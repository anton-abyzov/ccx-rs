# API & Core Loop Parity — Quick Scan Report

_Generated by research-agent scan of ccx-rs codebase._

---

## API Client (`crates/ccx-api`)

### What's Implemented

**SSE Event Types Handled** (`types.rs` `StreamEvent` enum):
- `MessageStart` ✅
- `ContentBlockStart` ✅
- `ContentBlockDelta` ✅ (TextDelta, InputJsonDelta, ThinkingDelta)
- `ContentBlockStop` ✅
- `MessageDelta` ✅ (with stop_reason + usage)
- `MessageStop` ✅
- `Ping` ✅ (parsed, ignored)
- `Error` ✅ (parsed, surfaced as `AgentLoopError::Api`)

**Prompt Caching Headers**: ❌ NOT present.  
The `stream_message` method sends `anthropic-version: 2023-06-01` and optionally `anthropic-beta: oauth-2025-04-20` but **never** sends `anthropic-beta: prompt-caching-2024-07-31`. The `Usage` struct does track `cache_creation_input_tokens` and `cache_read_input_tokens` (for parsing responses) but the request header to enable caching is missing.

**Extended Thinking**: ✅ Partially.  
`ContentBlock::Thinking`, `Delta::ThinkingDelta`, and `PendingBlock::Thinking` are all defined and handled in the stream consumer. However no `thinking` config is wired into `MessageRequest` (no `thinking: { type, budget_tokens }` field), so extended thinking cannot be *requested* — only parsed if returned.

**Error Types Handled** (`error.rs`):
- `Auth` — 401 Unauthorized
- `RateLimit { retry_after_secs }` — 429 with header parsing
- `Api { status, body }` — generic non-2xx
- `ApiEvent { error_type, message }` — SSE `error` event body
- `Overloaded` — HTTP 529
- `Http` — reqwest transport errors
- `Json` — serde_json errors
- `SseParse` — malformed SSE data

---

### API Client Gaps

1. **Prompt-caching beta header missing** — `anthropic-beta: prompt-caching-2024-07-31` never sent; caching won't activate.
2. **Extended thinking request field absent** — `MessageRequest` has no `thinking` field; can't request budget or type.
3. **`anthropic-beta` for extended thinking** — header `interleaved-thinking-2025-05-14` not sent.
4. **No `betas` field on `MessageRequest`** — can't pass arbitrary beta flags.
5. **`system` is `Option<String>`** — TS Claude Code sends system as a structured array with `cache_control` blocks (for prompt caching boundary). Rust uses a plain string.
6. **`MessageRequest` missing fields**: `top_p`, `top_k`, `metadata`, `stop_sequences`.
7. **`StopReason::PauseRun`** — not modeled (used in TS for plan mode / human-in-the-loop).
8. **`ContentBlock::ToolResult` accepts only `String` content** — TS supports structured content array (text + images).
9. **No image/base64 content block** — `ContentBlock` has no `Image` variant.
10. **`event_type` field parsed but ignored** — the SSE `event:` line is stored but not used for dispatch; routing is entirely by `type` field in JSON data (acceptable but fragile if types drift).

## Agent Loop (`crates/ccx-core/agent_loop.rs`)

### What's Implemented

**Loop Structure**: Classic query → stream → parse → tool dispatch → results → repeat.
- `send_message` appends user message, loops up to `max_turns` (default 50).
- Each turn: build `MessageRequest`, call `stream_with_retry`, collect `PendingBlock`s.
- If `stop_reason == ToolUse`, execute all tool calls and push results as a new user message, then continue loop.
- Terminates on `EndTurn` / `MaxTokens` / `StopSequence` / turn limit.

**Streaming Output Handling**:
- `consume_stream` processes events in real time.
- `on_text` callback fired per `TextDelta` chunk (live streaming to UI possible).
- `on_thinking` callback fired per `ThinkingDelta`.
- Tool input JSON buffered until `ContentBlockStop`.

**Tool Execution**: Sequential (one tool at a time, `for` loop over `tool_calls`).

**Stop Conditions**:
- `stop_reason != ToolUse` → return final text.
- `turn >= max_turns` → `AgentLoopError::MaxTurnsExceeded`.
- API error after retries → `AgentLoopError::Api` / `RateLimitExhausted`.

**Retry Logic** (`stream_with_retry`):
- Retries on `RateLimit` (uses `retry-after` header or exponential backoff).
- Retries on `Overloaded` (exponential backoff).
- Configurable via `RetryConfig` (max_retries=5, base=1s, cap=60s).
- `on_retry` callback notifies caller.

**Permission Gate**: `AgentCallback::should_allow_tool` — per-tool deny hook before execution.

---

### Agent Loop Gaps

1. **Sequential tool execution** — TS Claude Code dispatches all tools in a turn in **parallel** (`Promise.all`). Rust executes them one by one, which is significantly slower for multi-tool turns.
2. **No context limit / compaction** — SPEC Phase 4 mentions AutoCompact at ~187K tokens and MicroCompact (strip old tool results). Neither is implemented; long sessions will hit context limits silently.
3. **No token counting** — no tiktoken-rs integration; the loop cannot predict or preempt context overflow.
4. **`max_tokens` hardcoded to 16384** — not configurable per call or from config.
5. **No `stop_reason == MaxTokens` handling** — the loop exits (returns final text) rather than compacting and retrying.
6. **No `stop_reason == PauseRun`** — plan-mode / human-approval flow not modeled.
7. **`AgentManager` (`agent.rs`) is not wired into `AgentLoop`** — sub-agent spawning infra exists but the `Agent` tool (from SPEC P2-02) that would use it is absent.
8. **No CLAUDE.md / system prompt construction** — system prompt is a raw string passed in; SPEC P2-03 prompt crate not integrated.
9. **No MCP tool support** — `ToolRegistry` only handles static built-in tools.
10. **Conversation history grows unboundedly** — no trimming, summarisation, or windowing.
11. **No interrupt / cancellation signal** — no `CancellationToken` or ctrl-C handling in the loop.

## Auth (`crates/ccx-auth`)

### What's Implemented

**API Key Sources** (priority order):
1. Explicit key passed at call site
2. `ANTHROPIC_API_KEY` env var
3. macOS Keychain — `security find-generic-password -s "Claude Code-credentials"` ✅
4. `~/.claude/.credentials.json` fallback ✅
5. `~/.claude/config.json` (`apiKey` or `api_key` field) ✅

**OAuth Token Reuse**: ✅ Fully implemented.
- `AuthMethod::OAuthToken { access_token, subscription_type }` parsed from JSON key `claudeAiOauth`.
- Expiry check against `expiresAt` (ms epoch). Expired tokens silently skipped.
- Subscription type labels: `max`, `pro`, `team` → human-readable via `display_label()`.
- OAuth tokens sent as `Authorization: Bearer <token>` + `anthropic-beta: oauth-2025-04-20`.

---

### Auth Gaps

1. **No token refresh** — expired OAuth tokens are discarded; no refresh-token flow. User must re-login externally.
2. **No Linux/Windows keychain** — `read_keychain_token` is `cfg!(target_os = "macos")` only; other platforms fall through to file fallback.
3. **`resolve_api_key` ignores OAuth** — the older `resolve_api_key` function (still exported) skips keychain/credentials entirely; callers using it won't get OAuth.
4. **No `ANTHROPIC_AUTH_TOKEN` env var** — TS Claude Code also checks this alias; Rust only checks `ANTHROPIC_API_KEY`.
5. **No session token / STS support** — no `X-Amz-*` or Bedrock/Vertex auth paths.
6. **Config file path non-configurable** — hardcoded to `~/.claude/`; no `CLAUDE_CONFIG_DIR` override.
